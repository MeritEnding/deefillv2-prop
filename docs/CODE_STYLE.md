# 코드 스타일 가이드

읽기 쉽고 협업하기 쉬운 코드를 위한 팀 공통 규칙.
새 코드는 이 규칙을 따르고, 기존 코드는 **손대는 김에** 맞춘다 (보이스카우트 규칙 — 처음 왔을 때보다 깨끗하게).
Git 브랜치·커밋·PR 규칙은 레포 루트 `CLAUDE.md` 참고.

## 공통 원칙

1. **읽기 쉬움 > 영리함.** 한 줄 줄이려고 트릭을 쓰지 않는다. 6개월 뒤의 팀원이 바로 이해할 수 있어야 한다.
2. **함수는 한 가지 일만.** "~하고 ~한다"로 설명되면 쪼갠다. 화면 로직과 데이터 로직을 한 함수에 섞지 않는다.
3. **이름은 도메인 용어로.** `data2`, `tmp`, `flag` 금지. 탐지 결과는 `detections`, 마스크는 `mask`처럼 제품 용어를 그대로 쓴다.
4. **매직 넘버·문자열은 이름 있는 상수로.** 값의 의미(왜 이 값인지)를 상수 이름이나 주석으로 남긴다.
   예: `MASK_MARGIN = 0.25 // 탐지 박스 대비 마스크 타원 여유`
5. **같은 코드가 두 번 보이면 멈추고, 세 번째를 쓰기 전에 공통화한다.** 단, 우연히 닮은 코드(다른 이유로 변하는 코드)는 억지로 합치지 않는다.
6. **주석은 '왜'만 쓴다.** 코드가 말해주는 '무엇'을 반복하지 않는다. 제약·트레이드오프·비직관적인 이유를 적는다.
7. **죽은 코드는 지운다.** 주석 처리로 남기지 않는다 — 이력은 git에 있다.
8. **에러 메시지는 사용자 언어로.** 사용자에게 보이는 문구는 한글로, 원인과 다음 행동이 드러나게 쓴다.

## React (frontend/)

- **페이지는 조립, 로직은 lib.** `pages/`는 화면 구성과 상태만 담당한다. API 호출·이미지 처리 같은 재사용 로직은 `lib/`로 뺀다.
- **백엔드 호출은 반드시 `lib/api.js`를 통한다.** 컴포넌트 안에서 `fetch(API_BASE...)`를 직접 쓰지 않는다. 주소·에러 처리 방식이 한 곳에서 바뀌어야 한다.
- **화면에 뿌리는 고정 데이터는 컴포넌트 밖 대문자 상수로.** (`REDACT_ITEMS`, `KINDS` 패턴) 렌더 함수 안에서 배열을 새로 만들지 않는다.
- **상태는 최소한으로.** 다른 상태에서 계산되는 값은 state가 아니라 파생 변수로 둔다.
- **컴포넌트가 200줄을 넘으면 분리를 검토한다.** 시각적 블록(패널·목록·콘솔) 단위로 자른다.
- **props 이름은 받는 쪽 기준으로.** boolean은 `is-`/`has-` 또는 서술형(`auto`, `beta`), 콜백은 `on-` 접두어.

## Spring (backend/)

- **컨트롤러는 얇게.** 요청 검증 → 서비스/클라이언트 호출 → 응답 매핑까지만. 비즈니스 로직·상태는 서비스로.
- **에러 응답은 `@RestControllerAdvice` 한 곳에서.** 컨트롤러마다 try-catch로 JSON을 직접 만들지 않는다.
- **응답·헤더 이름은 상수 또는 한 곳에서 관리.** (`X-Engine`, `X-Elapsed-Ms` 등 프론트와 계약인 값)
- **설정값은 `@Value`/프로퍼티로 주입.** 코드에 키·주소를 하드코딩하지 않는다.

## Python (model/serving/)

- **공개 함수에는 타입 힌트와 한 줄 docstring.** 반환 형태(dict 구조 등)를 docstring에 적는다.
- **무거운 리소스(모델·캐스케이드)는 lazy 싱글턴 + lock.** import 시점에 로드하지 않는다. (`_get_detector()` 패턴)
- **엔드포인트 함수는 디코딩 → 처리 → 인코딩 흐름만.** 알고리즘은 별도 모듈(`detect.py` 등)로.

## 리팩토링 진행 규칙

- 리팩토링 PR은 **동작 변경과 섞지 않는다.** 기능 추가와 정리는 별 커밋, 가능하면 별 PR.
- 리팩토링 전후로 빌드(`npm run build`, `mvnw compile`)와 핵심 흐름(업로드→탐지→제거) 확인을 거친다.
- 공유 영역(다른 파트 소유 파일)을 정리할 때는 먼저 공지한다.
